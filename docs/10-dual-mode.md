# 10-dual-mode.md — 모의/실전 동시 운용(독립 2봇) + 상단 모드 토글

> 모의투자(paper)와 실전투자(live)를 **동시에 독립적으로** 운용한다. 상단 토글은
> "보는 모드"만 바꾸고, 양쪽 봇·시세는 백그라운드에서 계속 돈다(서버가 떠 있는 한).

## 1. 목표 / 범위

- 한 백엔드 프로세스에서 **모의 봇 + 실전 봇을 각각 on/off**로 독립 운용.
- 화면 최상단에서 **모의 ⇄ 실전 뷰 전환**. 전환은 표시 대상만 바꾸고 매매를 멈추지 않는다.
- **완전 모드별 독립**: 관심종목·전략·자본칸막이·잔고·주문·포지션·봇·시세 모두 모드별로 분리.
- 안전: 실전 시작은 명시적 확인 게이트, 실전 뷰는 시각 경고(빨간 배너), 일일 한도는 모드별 분리.
- 제약: 서버 재시작 시 인메모리 봇 상태는 초기화(둘 다 정지). "재시작 후 자동 재개"는 후속 과제.

## 2. 동시성 모델 (핵심 — 가장 중요한 설계)

- 런타임은 **asyncio 이벤트 루프(단일 스레드)**. 두 봇은 OS 스레드가 아니라 **비동기 태스크**다.
  → 전통적 race condition(메모리 동시 쓰기 손상)은 없지만, **격리는 공짜가 아니다.**
- 현재 코드의 **모드 무관 전역 공유 자원**을 모드별로 분리해야 진짜 독립이 된다:

| 전역 자원(현재) | 그대로 두면 | 분리 방식 |
| --- | --- | --- |
| `get_settings()` 전역 캐시 | 모드 1개 | 모드별 자격증명 접근 `settings.kis_for(mode)` |
| `KisAuth` 토큰 캐시 | 실전/모의 토큰 혼선 | **모드별 KisAuth 인스턴스**(토큰/도메인 분리) |
| `_rate_gate` 전역 리미터 | 모의(2/s)·실전(20/s)가 한 통에 묶여 상호 throttle | **모드별 레이트리미터** |
| `_shared_client` httpx(base_url 1개) | 모드별 도메인 달라 공용 불가 | **모드별 httpx 클라이언트** |
| `trading_bot`·`market_data_service` 전역 1개 | 봇 1개 | **모드별 인스턴스 레지스트리** |
| DB `orders.mode` | 이미 분리됨 ✅ | watchlist/strategies/budgets에도 `mode` 추가 |

## 3. 모드별 독립 범위 (데이터 컨텍스트)

전환 토글이 바꾸는 "모드 컨텍스트"는 다음을 **모드별로** 본다:

- **관심종목(watchlist)** · **전략(strategies)** · **자본칸막이(budgets)** — 테이블에 `mode` 컬럼 추가.
- **주문(orders)** — 이미 `mode` 컬럼 있음.
- **잔고/포지션/주문가능현금** — KIS 계좌가 모드별이라 본질적으로 분리(해당 모드 인증/계좌로 조회).
- **봇 상태/시세 수집 상태** — 모드별 인스턴스 상태.
- **매매 신호(signals)·감사 로그(audit)** — `mode` 태깅(표시 필터).

## 4. 설정 / 인증 (Phase 1)

- 자격증명은 **모드별 2세트**. 출처는 `Key/`(gitignore). 값은 `.env`(gitignore)에만 둔다.
- `.env` 키 레이아웃(신규):
  ```
  KIS_PAPER_APP_KEY / KIS_PAPER_APP_SECRET / KIS_PAPER_ACCOUNT_NO
  KIS_LIVE_APP_KEY  / KIS_LIVE_APP_SECRET  / KIS_LIVE_ACCOUNT_NO
  ```
  - 하위호환: 기존 `KIS_APP_KEY/SECRET/ACCOUNT_NO`가 있으면 **paper 기본값**으로 폴백.
- `Settings.kis_for(mode) -> KisCredentials`: `(app_key, app_secret, account_no, account_product, rest_base, ws_base, call_interval)`을 모드별로 반환.
- `trading_mode`는 이제 전역 스위치가 아니라 **기본 표시 모드(initial view)**.
- 실전 자격증명/계좌가 비면 실전 봇은 **구동 불가**(명확한 오류)로 둔다.

## 5. 런타임 분리 (Phase 2~3)

- 모드별 `KisAuth` / httpx 클라이언트 / 레이트리미터를 보유한 **`KisContext(mode)`** 도입(또는 동등 구조).
- KIS 호출 계층(client/auth/orders/account/quotes/charts/rankings)이 모드별 컨텍스트를 받도록 정리.
- **봇/시세 레지스트리**: `bots[mode]`, `feeds[mode]`. 봇 시작 시 해당 모드 시세도 보장.
  - 시세 데이터(체결가)는 모드 무관 동일하지만, **연결/인증이 모드별**이라 인스턴스도 모드별로 둔다(완전 독립 우선).

## 6. DB (Phase 4)

- `watchlist`, `strategies`, `budgets`에 `mode TEXT NOT NULL CHECK(mode IN ('paper','live'))` 추가.
- **마이그레이션**: 기존 행은 `paper`로 채운다(기존이 모의였으므로). WAL 유지.
- 모든 관련 쿼리·서비스가 `mode`로 필터.

## 7. API (Phase 5)

- 봇/시세/계좌/주문/관심종목/전략/칸막이/신호 라우터에 **`mode` 파라미터**(쿼리 또는 경로) 추가.
- `GET /api/bot/status`·`/api/market/status` 등은 모드별 상태 반환(또는 양쪽 요약).
- 실전 시작은 `confirm_live` 게이트 유지·강화.

## 8. 프론트 (Phase 6)

- 최상단 **모드 토글**(모의/실전). 선택 모드를 전역 상태로 두고 모든 패널이 그 모드 데이터를 조회.
- 토글은 표시 전환만; 봇 시작/정지는 각 모드 버튼에서. 양쪽 상태를 헤더에 함께 표기(예: "모의 봇 ●, 실전 봇 ○").
- **실전 경고**: 실전 뷰는 빨간 배너, 실전 봇 시작 시 확인 모달.

## 9. 안전

- 실전 봇 시작: 명시적 확인(이중 확인 모달) + 자격증명/계좌 존재 검증.
- 일일 주문 횟수/금액 한도는 **모드별** 집계(`orders.mode` 기준 — 이미 분리).
- 실전 자격증명 미설정 시 실전 시작 차단.
- 시크릿: `.env`/`Key/`만, 커밋 금지, 로그 마스킹 유지.

## 10. Phase 계획 (점진적 — 각 단계 테스트 통과 후 진행)

1. **Phase 1 — 설정/인증 분리**: `config.kis_for(mode)`, `.env` 모드별 키, 하위호환, 테스트. (실거래 트리거 없음)
2. **Phase 2 — 런타임 자원 분리**: 모드별 KisAuth/httpx/레이트리미터.
3. **Phase 3 — 봇/시세 레지스트리**: 모드별 인스턴스 + 라우터 `mode`.
4. **Phase 4 — DB 모드 컬럼**: watchlist/strategies/budgets 마이그레이션 + 서비스.
5. **Phase 5 — 프론트 토글**: 상단 전환 + 모드별 표시 + 실전 경고.
6. **Phase 6 — 통합 검증**: 모의/실전 봇 독립 동작, 뷰 전환 중 매매 유지(모의로 검증).

## 11. 테스트 전략

- 모드별 자격증명 라우팅(`kis_for`), 레이트리미터 모드별 분리.
- 두 봇 인스턴스가 서로의 상태/주문을 침범하지 않음(모드 격리).
- 뷰 전환 중에도 각 봇 `running` 유지.
- DB 마이그레이션: 기존 행 paper 채움, 모드별 쿼리 격리.
- 실전 시작 확인 게이트 / 자격증명 미설정 차단.
