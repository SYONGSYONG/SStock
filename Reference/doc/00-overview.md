## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  06-pages  →  05-conventions  →  04-tasks
   (왜)            (무엇)         (어떻게)    (어떻게 보이는가)  (어디에)      (어떤 규칙)       (무엇을 완료했나)
```

**Claude Code는 작업 시작 전 이 순서대로 6개 파일을 모두 읽는다.**
앞 문서를 이해해야 뒷 문서의 맥락이 성립하므로 순서를 건너뛰지 않는다.

---

## 왜 6개 파일로 나누는가 — 관심사 분리(Separation of Concerns) 원칙

WinForm 개발에 비유하면, 하나의 `.cs` 파일에 UI 코드·비즈니스 로직·DB 쿼리를 모두 넣지 않는 것과 같다.
각 파일은 **하나의 질문에만 답한다**. 섞이는 순간 수정 시 다른 영역까지 영향을 받는다.

| 질문 | 담당 파일 | 섞이면 생기는 문제 |
|------|----------|--------------------|
| 무엇을 만드는가? | `01-product` | 기능 변경이 기술 명세를 오염시킴 |
| 어떤 기술로 만드는가? | `02-specs` | 스택 교체 시 제품 정의까지 수정해야 함 |
| 어떻게 보이는가? | `03-design` | 디자인 변경이 페이지 로직에 영향 |
| 무엇을 언제 만드는가? | `04-tasks` | 단계 순서가 흐트러지면 검증 누락이 생김 |
| 어떤 규칙을 따르는가? | `05-conventions` | 규칙이 분산되어 일관성 깨짐 |

결론: **문서를 나누는 것은 코드를 모듈로 나누는 것과 동일한 이유다.**
변경 범위를 최소화하고, 필요한 정보를 빠르게 찾기 위함이다.
