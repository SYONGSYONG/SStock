#   — 제품 정의

## 제품 목표

1. **레포지토리 상태 한눈에 확인** — 슈퍼 레포지토리(Framework5.0)와 N개 서브 레포지토리의 브랜치·커밋 현황을 하나의 대시보드에서 읽기 전용으로 파악한다.
2. **이슈 한눈에 확인** — 각 레포지토리에 등록된 이슈의 진행 상태를 대시보드에서 즉시 확인한다.

## 페르소나

| 항목 | 내용 |
|------|------|
| 직책 | 14인 소프트웨어 팀의 인프라 구축 담당자 (과장) |
| 주요 관심사 | 전체 레포지토리 상태 파악, 이슈 배분 및 진행 현황 추적 |
| 팀원 역할 | 누구든 이슈를 **등록**할 수 있고, **담당자**가 될 수 있다 |
| 팀 규모 | 14명 (슈퍼 레포 관리자 1인 + 팀원 13인) |

## MVP 범위

### 기능 1 — 레포지토리 현황 대시보드

- 슈퍼 레포지토리와 서브 레포지토리 목록 표시
- 이름 / 마지막 커밋 날짜 기준 오름차순·내림차순 정렬
- 레포 행 클릭 시 **팝업** 열림:
  - **README.md 뷰어** — Markdown 렌더링, 상대 링크 → GitHub 절대 URL 자동 변환
  - **커밋 히스토리** (최신 10건) — 클릭 시 커밋 상세 (SHA, 작성자, 변경 파일 목록) 표시
  - **ZIP 다운로드** 버튼 — 확인 다이얼로그 후 main 브랜치 ZIP 다운로드
  - **GitHub 열기** 버튼 — 확인 다이얼로그 후 GitHub 페이지 새 탭 오픈
  - **Clone** 버튼 — 확인 다이얼로그 + 경로 입력 후 백엔드에서 `git clone` 실행
- 레포지토리는 **읽기 전용** 접근만 허용. 파일 추가·커밋·브랜치 생성 불가

### 기능 2 — 이슈 게시판 / To Do 게시판

이슈 게시판과 To Do 게시판은 **전체 레포지토리를 통틀어 하나씩** 존재한다. 두 게시판은 동일한 UI를 사용하되 `type` 필드로 구분된다. 이슈/To Do는 특정 레포에 종속되지 않으며, 칸반 등록을 통해 개별 레포와 연결된다.

#### 이슈/To Do 공통 필드

| 필드 | 필수 여부 | 설명 |
|------|----------|------|
| 제목 | ✅ 필수 | 이슈/To Do 요약 |
| 상세 내용 | ✅ 필수 | 본문 |
| 의뢰자 | ✅ 필수 | 등록한 팀원 이름 (자유 텍스트 입력) |
| 등록일 | ✅ 필수 | 등록 시각 (자동 입력) |
| 담당자 | 선택 | 처리할 팀원 (1인, 관리자 등록 멤버에서 선택) |
| 완료일 | 선택 | 실제 완료된 날짜 |
| 난이도 | 선택 | 작업 난이도 (LOW / MEDIUM / HIGH) |
| 구분 | 자동 | `issue` 또는 `todo` |

#### 게시판 부가 기능

- **처리 상태 표시** — 이슈 목록에서 현재 칸반 상태(TO DO / DOING / DONE / PENDING) 확인 가능
- **댓글** — 이슈 상세 페이지에서 팀원 누구나 댓글 등록 가능. 작성자를 멤버 목록에서 선택하며, 미선택 시 익명으로 저장됨
- **칸반 등록** — 이슈 상세에서 담당자를 지정하고 칸반의 `TO DO`로 직접 이동 가능 (복수 레포 동시 등록 가능)
- **자동 댓글** — 칸반 상태 이동 시 이슈에 자동 댓글 등록 (담당자 멘션 포함)
- **필터 + 검색** — 레포지토리 필터, 상태 필터, 텍스트 검색(제목 / 제목+내용 / 의뢰자 / 담당자) 지원
- **이슈 삭제** — 목록에서 행 호버 시 삭제 버튼 노출. 삭제 시 관련 댓글·칸반 기록도 함께 제거
- **종결 / 되살리기** — 이슈 상세에서 종결 버튼(DONE 또는 PENDING 상태일 때만 활성). 종결 시 모든 칸반에서 제거. 되살리기 시 미등록 상태로 복원 (칸반 복원 없음)

### 기능 3 — 칸반 보드

#### 칸반 카드 표시

- 카드에 **제목 + 담당자** 표시
- To Do 항목은 **emerald 배경 + TD 뱃지**로 이슈와 구분
- 통합 뷰에서는 레포별 색상 테두리 + 레포명 태그 표시
- 복수 레포에 등록된 이슈는 연관 레포 태그 표시
- **카드 클릭 시 팝업** (읽기 전용): 이슈 정보 + 최근 댓글 표시. 게시판 이동 버튼 포함
- **이슈/To Do 타입 필터 탭** (전체 / 이슈 / To Do)

#### 칸반 컬럼

| 컬럼 | 의미 |
|------|------|
| `TO DO` | 작업 예정 |
| `DOING` | 진행 중 |
| `DONE` | 완료 |
| `PENDING` | 보류 |

#### 허용된 상태 전환

```
TO DO   →  DOING
DOING   →  DONE
DONE    →  DOING     ← 재작업 필요 시
TO DO   →  PENDING
DOING   →  PENDING
DONE    →  PENDING
PENDING →  TO DO     ← 보류 해제 후 재시작
PENDING →  DOING     ← 보류 해제 후 즉시 진행
PENDING →  DONE      ← 보류 해제 후 완료 처리
```

> PENDING은 어느 상태로든 전환 가능하다. 보류 해제 시 이동할 상태를 명시적으로 선택한다.

#### 이슈 ↔ 칸반 다대다 관계

- 하나의 이슈는 **N개의 레포지토리 칸반**에 동시에 등록될 수 있다.
  (이유: 하나의 이슈가 복수 레포지토리의 수정을 수반할 수 있기 때문)
- 복수 칸반에 등록된 이슈는 **어느 칸반에서 상태를 변경해도 전체 동기화**된다.
  - 예: 이슈 #42가 `repo-A` 칸반과 `repo-B` 칸반에 등록된 경우,  
    `repo-A` 칸반에서 `DOING → DONE`으로 이동하면 `repo-B` 칸반도 `DONE`이 된다.

#### 슈퍼 레포지토리 칸반의 통합 뷰

- 슈퍼 레포지토리 칸반은 **자신의 이슈 + 모든 서브 레포지토리의 이슈**를 함께 표시한다.
- 슈퍼 레포지토리 칸반에서 서브 레포지토리 이슈의 **상태 전환이 가능**하다.
- 슈퍼 레포지토리 칸반에서의 상태 변경도 동일하게 **전체 동기화**된다.
  - 예: `repo-A` 이슈 #7이 슈퍼 레포 칸반에 표시된 상태에서,  
    슈퍼 레포 칸반에서 `TO DO → DOING`으로 이동하면 `repo-A` 칸반도 `DOING`이 된다.
- 슈퍼 레포지토리 칸반은 출처 레포지토리를 구분할 수 있도록 **레포 이름 태그**를 카드에 표시한다.

## 자동 댓글 규칙

칸반 상태 전환이 발생하면 해당 이슈 게시판에 **시스템 댓글**이 자동 등록된다.

| 이벤트 | 자동 댓글 형식 예시 |
|--------|-------------------|
| TO DO → DOING | `[repo-A] @홍길동 님이 작업을 시작했습니다. (TO DO → DOING)` |
| DOING → DONE | `[repo-A] @홍길동 님이 작업을 완료했습니다. (DOING → DONE)` |
| DONE → DOING | `[repo-A] @홍길동 님이 재작업을 시작했습니다. (DONE → DOING)` |
| → PENDING | `[repo-A] @홍길동 님에 의해 보류 처리되었습니다. (DOING → PENDING)` |
| PENDING → TO DO | `[repo-A] @홍길동 님이 보류를 해제하고 대기 상태로 전환했습니다. (PENDING → TO DO)` |
| PENDING → DOING | `[repo-A] @홍길동 님이 보류를 해제하고 작업을 재개했습니다. (PENDING → DOING)` |
| PENDING → DONE | `[repo-A] @홍길동 님이 보류를 해제하고 완료 처리했습니다. (PENDING → DONE)` |

> 슈퍼 레포지토리 칸반에서 전환 시 `[repo-A]` 자리에 출처 레포지토리 이름이 표시된다.

## 회원관리 및 인증 기능

### 회원 필드

| 필드 | 필수 여부 | 설명 |
|------|----------|------|
| 이름 | ✅ 필수 | 표시 이름 |
| 소속 | ✅ 필수 | 소속 부서/팀 |
| 직급 | ✅ 필수 | 직급/직책 |
| 전화번호 | ✅ 필수 | 연락처 |
| 이메일 | ✅ 필수 | 로그인 ID로 사용, 유니크 |
| 비밀번호 | ✅ 필수 | bcrypt 해시 저장. 8자 이상 + 영문자(대소문자 무관) 1자 이상 + 숫자 1자 이상, 특수문자·공백 불가 |
| 역할 | 자동 | `member` (기본) / `team_member` / `admin` |

### 회원 역할(Role) 구분

| 역할 | 설명 | 권한 |
|------|------|------|
| `member` | 일반 회원 (신규 가입 기본) | 이슈/댓글 조회·등록, 자신의 글 삭제 |
| `team_member` | 팀원 | member 권한 + 칸반 상태 이동 |
| `admin` | 관리자 | 모든 권한 + 회원 관리 (역할 변경, 비밀번호 초기화, 삭제) |

### 초기 계정

- 초기 계정: **MASTER** (name='MASTER', email='master@admin.local', password='1234', role='admin')
- 앱 최초 시작 시 자동 생성

### 인증 흐름

- 모든 페이지는 로그인 필수 (미인증 시 `/login` 리다이렉트)
- 로그인: 이메일 + 비밀번호 → JWT 발급 (24시간 유효)
  - 계정이 승인 대기(`is_active=0`)인 경우 403 `PENDING_APPROVAL` 반환, 로그인 화면에 "관리자 승인 대기 중" 안내 표시
- 회원가입: 이름·소속·직급·전화번호·이메일·비밀번호 입력 → 역할 `member`, `is_active=0`(승인 대기) 상태로 생성
  - 가입 직후에는 로그인 불가. 관리자가 회원관리 페이지에서 승인하면 활성화됨
- 비밀번호 초기화 요청: 이메일 입력 → `password_reset_requested=1` 플래그 설정
  - 관리자가 회원관리 페이지에서 확인 후 `1234`로 초기화

### 회원 연동 규칙

| 기능 | 연동 내용 |
|------|----------|
| 이슈/To Do 등록 | 의뢰자 = 로그인 사용자 (자동, 입력 불필요) |
| 댓글 등록 | 작성자 = 로그인 사용자 (자동, 선택 콤보박스 제거) |
| 칸반 상태 이동 | `team_member` 또는 `admin`만 가능 |
| 이슈/To Do 삭제 | 작성자 본인 또는 `admin`만 가능 |
| 댓글 삭제 | 작성자 본인 또는 `admin`만 가능 (댓글 삭제 기능 신규 추가) |

### 회원관리 페이지 (`/members`, 기존 `/admin` 대체)

- **회원가입 승인 대기 섹션** (대기 회원 ≥ 1명일 때) — 카드형 접힘/펼침 UI
  - 접힌 상태: `이름 : OOO, 소속 : OOO` + 우측 [승인][거절] 버튼
  - 펼친 상태: 이름·소속·직급·전화번호·이메일·신청일 상세 정보
  - 승인 → `is_active=1`로 전환, 거절 → 회원 삭제
- **비밀번호 초기화 요청 섹션** (요청 ≥ 1건일 때) — 요청자별 [1234로 초기화] [거절] 버튼
- **전체 회원 목록 테이블** — 역할 변경, 비밀번호 초기화, 삭제. 승인 대기 회원은 행 opacity dim + "대기" 뱃지 표시
- 관리자(`admin`) 전용 페이지 (`AdminRoute` 보호, 비관리자 접근 시 `/`로 리다이렉트)

### 활동 로그 페이지 (`/logs`, admin 전용)

- 시스템에서 발생하는 모든 주요 동작을 텍스트 파일(`backend/logs/activity.log`)에 기록
- 로그 카테고리: `LOGIN`, `LOGOUT`, `ISSUE`, `COMMENT`, `KANBAN`, `MEMBER`
- 기록 이벤트: 로그인 성공·실패, 로그아웃, 회원가입, 비밀번호 변경, 비밀번호 초기화 요청, 프로필 수정, 회원 승인·역할 변경·삭제, 이슈/To Do CRUD, 댓글 추가·삭제, 칸반 등록·해제·상태 변경
- UI: 카테고리별 색상 뱃지, 카테고리 필터, 줄 수 선택(100~2000), 10초 자동 새로고침, 최신순(역순) 정렬
- 로그 파일 10MB 초과 시 자동 로테이션 (날짜·시각 suffix가 붙은 아카이브 파일로 분리)
